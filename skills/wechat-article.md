# 위챗 공식계정 글: 작가-편집자-독자 3-Agent 협업

$ARGUMENTS 에 대해 심층 리서치를 수행하여, 바로 발행 가능한 위챗 공식계정 글 한 편을 만들어낸다. 세 개의 Agent가 각자의 역할을 맡는다: 작가는 깊이 있는 초고를 쓰고, 편집자는 구조와 표현을 다듬으며, 독자는 타깃 독자의 시각에서 검토한다.

**지원 입력 형식**: 주제 설명, 예: `대규모 모델 OPD 기술 해설`, `Qwen3 기술 보고서 해설`, `왜 Warren Buffett(버핏)은 기술주를 사지 않는가`

---

## 설계 철학

좋은 공식계정 글은 세 가지 차원을 동시에 만족해야 한다:
1. **깊이**——시간을 들여 끝까지 읽는 사람에게 값어치를 한다(작가 담당)
2. **가독성**——구조가 명확하고 리듬이 좋아 도중에 이탈하지 않게 한다(편집자 담당)
3. **정말 이해되는가**——타깃 독자가 중간에 포기하지 않는다(독자 담당)

혼자 쓰면 "자아도취"에 빠지기 쉽다——쓰는 사람은 명확하다고 느끼지만 읽는 사람은 이해하지 못한다. 3-Agent 협업의 본질은 **외부 시각을 강제로 끌어들이는 것**이다.

---

## 1단계: 리서치와 자료 수집

### 첫째 단계: 글의 포지셔닝 명확화

글쓰기를 시작하기 전에 다음 정보를 먼저 확인한다(사용자가 지정하지 않았으면 먼저 물어본다):

| 차원 | 확인할 사항 | 기본값 |
|------|---------|--------|
| **타깃 독자** | 기술적 배경 수준 | 어느 정도 기술 배경은 있으나 해당 분야 전문가는 아님 |
| **글의 깊이** | 입문/중급 심화/하드코어 | 중급 심화(공식은 있되 알기 쉽게 설명) |
| **글의 길이** | 글자 수 범위 | 3000~4000자 |
| **원본 논문/자료 다운로드 필요 여부** | PDF/도판 필요 | 예 |
| **글쓰기 스타일** | 격식체/대화체/예리함 | 대화체(똑똑한 친구에게 쓰듯이) |

### 둘째 단계: 심층 리서치

Agent 도구를 사용해 2~3개의 리서치 Agent를 **병렬로** 띄워 충분한 자료를 수집한다:

**리서치 Agent A: 핵심 내용 리서치**
- 논문 해설이라면: 논문 PDF 다운로드, 핵심 기여·주요 도표·실험 결과 추출
- 기술 주제라면: 최신 진전·핵심 논문·기술 세부 사항 검색
- 비즈니스/투자 주제라면: 최신 데이터·산업 보고서·경쟁 구도 검색

**리서치 Agent B: 산업 배경과 응용**
- 해당 기술/주제의 산업 현장 적용 상황 검색
- 어떤 기업들이 쓰고 있는가? 효과는 어떤가?
- 최신 발전 트렌드와 이정표적 사건

**리서치 Agent C(선택): 경쟁 제품/비교 리서치**
- 동종 방법/제품의 비교
- 역사적 발전 흐름
- 향후 진화 방향

### 셋째 단계: 자료 프레임 정리

리서치 Agent가 모두 완료되면 다음을 정리한다:
1. **핵심 논점**(글이 전달하려는 핵심 메시지를 한 문장으로 요약)
2. **핵심 데이터**(가장 충격력 있는 데이터 포인트 3~5개)
3. **도판 목록**(어떤 그림이 필요하며 출처는 무엇인가)
4. **글의 개요**(6~8개 장(章)의 제목과 핵심 내용)

## 2단계: 작가 Agent가 초고 작성

Agent 도구로 **작가 Agent**를 띄우고 상세한 작성 지침을 준다.

### 작가 Agent의 Prompt 템플릿

```
당신은 심층 기술 작가(Author Agent)로, 위챗 공식계정 글 한 편을 써야 한다.

## 타깃 독자
{첫째 단계에서 확인한 독자 프로필}

## 글쓰기 스타일 요구
- 순 한국어 표현, 한영 혼용을 피한다(기술 용어는 처음 등장할 때 영어를 병기하고 이후에는 한국어 사용)
- 학술 논문 번역이 아니라, 똑똑한 친구에게 보여주는 기술 해설처럼
- 비유로 이해를 돕되, 비유는 적확하고 진부하지 않게
- 핵심 공식/데이터는 넣되, 각각을 쉬운 말로 설명할 것
- 이모지 사용 금지
- 단락은 4줄을 넘지 않는다(공식계정 읽기 환경 고려)

## 핵심 내용
{정리된 자료, 데이터, 논점}

## 글 구조 요구
1. **도입(앞 3개 단락)**: 반드시 강한 후크가 있어야 한다——데이터의 충격력이나 반직관적인 결론으로 시작하고, 부드러운 비유로 시작하지 말 것
2. **배경**: 왜 이 일이 중요한가? 어떤 문제를 푸는가?
3. **핵심 내용(2~3절)**: 기술적 깊이는 여기서 드러나되, 모든 기술 포인트에는 "쉬운 말 번역"이 있어야 한다
4. **실증/사례**: 데이터와 사례로 말하고, 공허한 이야기는 하지 말 것
5. **산업 영향/전망**: 이 일이 산업에 무엇을 의미하는가
6. **결말**: 전파력 있는 한 문장의 판단으로 마무리하고, 캡처해서 공유하기 좋게

## 도판 요구
- 논문 해설류 글: 반드시 논문 PDF에서 원본 그림을 추출하여 `![설명](상대경로)` 로 글에 직접 삽입하고, [그림X: 설명] 같은 플레이스홀더를 쓰지 말 것
- 추출 방법: pdftoppm 로 PDF 페이지를 고해상도 PNG로 렌더링(최소 900 DPI)한 뒤 PIL로 대상 도표 영역을 잘라낸다
- 각 그림은 500KB 이상으로 하여 고화질을 보장할 것
- 이미지는 `assets/{주제 약칭}/` 디렉토리에 통일해 보관
- 비논문류 글: 도판이 필요하면 적절한 이미지를 검색·다운로드하여 마찬가지로 직접 삽입

## 공식 요구
- 모든 수학 공식은 LaTeX 형식 사용: 인라인은 `$...$`, 독립 공식은 `$$...$$`
- 순수 텍스트로 공식을 쓰는 것(예: `> D_KL(P || Q) = ...`) 금지, 반드시 LaTeX 렌더링 형식 사용
- 각 공식 뒤에도 "쉬운 말 번역"을 붙일 것

완전한 글 초고를 약 {목표 글자 수}자로 작성하라.
```

### 작가 Agent 완료 후

초고 파일이 생성되었는지 확인하고, 전문을 읽어 내용의 완결성을 점검한다.

## 3단계: 편집자 Agent + 독자 Agent 병렬 검토

초고가 완성되면 Agent 도구로 **같은 메시지 안에서** 편집자 Agent와 독자 Agent를 동시에 띄운다.

### 편집자 Agent의 Prompt 템플릿

```
당신은 베테랑 공식계정 편집자(Editor Agent)다. 다음 글을 정밀하게 검토하라.

## 검토 기준
1. **제목**: 피드(타임라인)에서 클릭을 끌 수 있는가? 잘리지 않는가(30자 초과)?
2. **도입**: 앞 3개 단락이 독자를 붙잡을 수 있는가? 후크가 충분히 강한가?
3. **구조**: 논리 사슬이 매끄러운가? 비약이나 단절은 없는가?
4. **깊이와 가독성의 균형**: 공식/기술 부분이 정말 쉬운가? "쉬운 척하지만 제대로 설명 안 한" 곳은 없는가?
5. **리듬**: 너무 긴 단락은 없는가? 각 절의 길이는 적절한가?
6. **도판**: 이미지가 실제로 삽입되어 있는가(플레이스홀더가 아닌)? 독자가 시각적 보조가 가장 필요할 때 위치하는가?
7. **결말**: 전파력이 있는가? 다 읽은 독자가 공유하고 싶어지는가?

## 글 전문
{완성된 초고}

## 출력 형식
1. 총평(3~5문장)
2. 제목 수정 제안(2~3개 대안 제시)
3. 절별 수정 제안("원문 → 이렇게 수정 권장"의 구체적 대조 제시)
4. 가장 중요한 개선 포인트 3가지
```

### 독자 Agent의 Prompt 템플릿

```
당신은 {타깃 독자 프로필}(Reader Agent)이다. 독자 시각에서 다음 글을 검토하라.

## 당신의 배경
{타깃 독자의 지식 수준과 읽기 습관을 구체적으로 기술}

## 글 전문
{완성된 초고}

## 다음 질문에 답하라
1. 앞 3개 단락을 읽고 나서 계속 읽고 싶은가? 이유는?
2. 어느 부분이 "이해 안 되거나" "다시 읽어야 이해되는가"? 구체적으로 어느 문장인가?
3. 기술/공식 부분을 이해했는가? "쉬운 말 번역"이 도움이 되었는가?
4. 글의 핵심 비유가 적확한가? 더 나은 비유가 있는가?
5. 너무 긴가 짧은가? 어디에서 인내심을 잃게 되는가?
6. 다 읽고 나서 글의 핵심 관점을 한 문장으로 요약할 수 있는가?
7. 이 글을 공유하겠는가? 공유할 때 뭐라고 말하겠는가?
8. 알고 싶었지만 글이 다루지 않은 질문이 있는가?
```

## 4단계: 최종 원고 확정

### 첫째 단계: 두 Agent의 피드백 종합

다음과 같은 빈출 문제에 집중한다:

| 문제 유형 | 편집자 흔한 피드백 | 독자 흔한 피드백 | 처리 방식 |
|---------|------------|------------|---------|
| 도입이 약함 | 후크가 부족 | 앞 3단락에서 계속 읽을 동기가 없음 | 데이터/반직관 결론으로 도입을 다시 씀 |
| 기술 단락이 이탈 유발 | 공식이 너무 빽빽함 | 어떤 단락을 3번 다시 읽어야 함 | 공식 삭제 또는 그림화, 더 직관적인 비유 추가 |
| 리듬이 늘어짐 | 어떤 절이 너무 김 | 어디선가 인내심을 잃음 | 통합 또는 삭감(특히 후반부의 반복적인 기술 설명) |
| 결말이 약함 | 전파력 부족 | 공유하지 않을 것 | 캡처해 전파 가능한 한 문장의 판단으로 다시 씀 |
| 개념 비약 | 논리에 단절 있음 | 어디선가 "갑자기 이해 안 됨" | 전환 문장이나 배경 설명 보충 |

### 둘째 단계: 수정 실행

피드백에 따라 글을 다시 쓴다. 핵심 수정 원칙:

1. **편집자와 독자가 모두 지적한 문제는 반드시 고친다**
2. **편집자만 지적한 문제는 고칠 가능성이 크다**(편집자의 전문 판단은 대개 정확하다)
3. **독자만 지적한 문제는 상황에 따라 고친다**(독자 피드백은 실제 경험을 대변하지만 모든 항목에 반드시 대응할 필요는 없다)
4. **둘이 충돌할 때는 독자 쪽으로 기운다**(편집자는 완벽을 추구하지만 독자 경험이 최종 기준이다)

### 셋째 단계: 도판 추출

논문 해설류 글은 최종 원고 확정 전에 반드시 도판 추출을 완료한다:

1. **렌더링**: `pdftoppm -png -r 900 -f {페이지} -l {페이지} 논문.pdf /tmp/page` (900 DPI에서 시작, 이미지가 500KB에 못 미치면 1200 또는 1500 DPI로 올림)
2. **위치 확인**: 먼저 150 DPI로 전체 페이지를 렌더링하여 각 도표의 픽셀 좌표를 눈으로 확인
3. **잘라내기**: PIL로 좌표에 따라 잘라내고 `compress_level=1` 로 저장하여 각 그림 ≥ 500KB 보장
4. **저장**: `assets/{주제 약칭}/` 디렉토리에 저장, 파일명은 `fig{번호}-{설명}.png`
5. **삽입**: 글에서 `![설명](../../assets/{주제 약칭}/fig{번호}-{설명}.png)` 로 참조

### 넷째 단계: 최종 파일 산출

최종 원고를 md 파일로 저장하고, 파일 끝에 논문/자료 원문 링크를 첨부한다:

```markdown
**논문 원문:**
- arXiv: {링크}
```

## 글쓰기 레드라인

1. **데이터를 지어내지 않는다**. 인용 데이터는 반드시 출처가 있어야 하고, 찾을 수 없으면 "추정"이라 표기
2. **AI 말투를 쓰지 않는다**. "함께 한번 살펴봅시다", "주목할 만한 점은", "말하지 않을 수 없는 것은" 같은 상투어 금지
3. **과도한 약속을 하지 않는다**. 기술 글에서 "파괴적", "혁명적" 같은 말을 쓰지 말고 데이터로 말한다
4. **공식에는 반드시 쉬운 말을 붙인다**. 각 공식 뒤에 "사람 말로 옮기면……" 한 단락을 둔다
5. **공식은 반드시 LaTeX로**. `$$...$$` 형식, 순수 텍스트 공식 금지
6. **도판은 반드시 실제로 삽입한다**. 논문 해설류는 PDF에서 고화질 원본 그림(≥500KB)을 추출하고, `[그림X]` 플레이스홀더 금지
7. **표의 괄호 주석은 정확하게**. 개념을 설명할 때 정확한 정의를 쓰고, 모호한 동목 구문을 쓰지 않는다(예: "교사를 배운 텍스트"가 아니라 "교사로부터 온 텍스트")
8. **비유는 일관되게**. 글 전체에 하나의 주축 비유를 쓰고, 절마다 새 비유로 바꾸지 않는다
9. **결말에는 반드시 전파력이 있어야 한다**. 마지막 문장은 단독으로 캡처해 공유할 만한 값어치가 있어야 한다
