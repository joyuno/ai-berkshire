# 기업 뉴스 펄스: 주가 급변 신속 원인규명 팀

$ARGUMENTS 에 대해 최근 뉴스 정찰과 급변 원인규명을 수행한다. **이것은 심층 투자 리서치가 아니라 정보의 신속 대응이다.** 목표는 10분 안에 다음 질문에 답하는 것이다. "이 회사에 최근 무슨 일이 있었나? 주가 급변의 진짜 원인은 무엇인가? 투자 논거를 다시 검토해야 하나?"

## 적용 상황

- 보유/관심 종목의 주가가 급등/급락했을 때 (일반 트리거 기준: 일간 ±5%, 주간 ±10%)
- 실적 발표 후 주가가 급변해, 시장이 무엇에 반응하는지 빠르게 파악하고 싶을 때
- 뉴스 헤드라인을 봤지만 노이즈인지 진짜 시그널인지 확신이 서지 않을 때
- **부적합한 경우**: 완전한 투자 리서치(`/investment-team` 사용), 실적 심층 분석(`/earnings-review` 사용), 장기 투자 논거 추적(`/thesis-tracker` 사용)

## 실행 절차

### 1단계: 파라미터와 상황 확인

다음 정보를 사용자에게 명확히 확인한다 ($ARGUMENTS 에 제공되지 않은 경우).

| 파라미터 | 설명 | 기본값 |
|------|------|------|
| **회사명** | 중문/영문/종목코드 모두 가능 | 필수 |
| **시간 범위** | 뉴스를 거슬러 정찰할 일수 | 기본 14일, 실적 시즌에는 7일로 단축 가능 |
| **주가 급변** | 등락폭 + 기간, 예: "12% 하락/3일" | 선택, 있으면 원인규명을 집중하는 데 사용 |
| **관심 비중** | 기업 이벤트 / 규제 / 업계 / 심리 | 기본은 4개 균등 |

사용자가 회사명만 말했다면, 먼저 되묻는다. "최근 며칠치 뉴스를 볼까요? 설명이 필요한 구체적인 주가 급변이 있나요?" — **조용히 가정하지 말 것.**

### 2단계: 정보 가용성 등급 분류

`investment-team.md` 의 A/B/C 등급을 참고하되, 기준 차원은 다르다.

| 등급 | 특징 | 정찰 전략 |
|------|------|---------|
| **A급 (정보 풍부)** | 대형주, 미디어 커버리지 넓음, 실적 시즌 | 핵심은 **노이즈 제거와 원인규명** — 정보가 너무 많으면 오히려 진짜 원인을 찾기 어렵다. 각 Agent는 판단력을 갖춰 "반복 전달"되는 2차 뉴스를 걸러내야 한다 |
| **B급 (정보 중간)** | 중소형주, 커버리지 보통 | 표준 모드, 핵심 이벤트마다 1~2개 독립 출처를 첨부 |
| **C급 (정보 희소)** | 홍콩주 소형주, 신규 상장, 비인기주 | "기초 탐색 모드"로 전환 — 급변을 설명할 뉴스를 전혀 못 찾을 수도 있는데, **이 결론 자체가 가치가 있다**(기본 펀더멘털이 아니라 기술적/수급 요인일 가능성) |

등급은 각 Agent에게 알려 정찰 방식에 반영되도록 한다.

### 3단계: 팀 생성

TeamCreate 로 팀을 생성한다.
- `team_name`: `{회사명}-newspulse` (영문 소문자, 예: `pdd-newspulse`)
- `agent_type`: `team-lead`

### 4단계: 4개의 정찰 작업 생성

TaskCreate 로 다음 4개 작업을 생성한다.

#### 작업 1: 기업 이벤트 정찰 (company-event-scout)

- **subject**: `{회사명}의 최근 {N}일간 기업 자체 이벤트 정찰`
- **description**:
  1. **공식 공시**: 홍콩거래소/SEC/쥐차오(巨潮) 등 규제 공시 플랫폼의 최근 공시
  2. **실적과 가이던스**: 최신 분기/연간 보고서, 실적 예고, 실적 발표회 핵심 내용
  3. **경영진 동향**: 임원 변동, 지분 증감, 자사주 매입, 배당, 스톡옵션
  4. **중대 사업 이벤트**: 신제품 출시, 인수합병/구조조정, 사업 매각, 대형 고객/대형 수주
  5. **자본 운용**: 추가 증자, 전환사채, ADR 전환, A주 복귀/상장폐지 동의안
  6. **소송과 컴플라이언스**: 피소, 자체 공시한 컴플라이언스 사안
  7. 각 이벤트에 표기: **날짜 / 출처 링크 / 한 줄 요약 / 주가 급변과의 상관 가능성(고/중/저)**
  8. 시간 역순 타임라인 표로 출력

#### 작업 2: 규제와 정책 (regulatory-watcher)

- **subject**: `{업종/회사}의 최근 {N}일간 규제와 정책 변화 정찰`
- **description**:
  1. **업종 규제**: 소속 업종의 신규 규정, 벌금, 시정 명령, 라이선스 변동
  2. **국경 간 정책**: 미중 관계(중국 개념주), 관세, 수출 통제, 데이터 안보
  3. **세제 정책**: 부가가치세, 법인세, 소득세 관련 변화
  4. **반독점과 경쟁법**: 조사, 벌금, 인수합병 불허
  5. **특수 업종 정책**: 의약품 집중구매, 교육 쌍감(双减), 부동산 3대 레드라인, 인터넷 플랫폼 규제 등
  6. **통화와 외환**: 해당 기업에 영향을 주는 환율/금리/자본 통제 변화
  7. 각 정책에 표기: **날짜 / 출처 / 해당 기업에 미치는 직접 영향 정도(직접/간접/무관)**
  8. 핵심 판단: "정책 블랙스완"이 방금 현실화되었는지 여부

#### 작업 3: 업계와 경쟁사 (industry-peer-analyst)

- **subject**: `{회사명}의 업계 구도와 경쟁사의 최근 {N}일간 동향 정찰`
- **description**:
  1. **직접 경쟁사**: 핵심 경쟁사 3~5개를 나열하고, 각각의 최근 이벤트(실적, 제품, 가격 경쟁, 인사)를 조사
  2. **밸류체인 전후방**: 상류 원자재/공급업체, 하류 고객/유통 채널의 최근 가격·생산능력·수주 변화
  3. **업계 전반**: 업황 데이터, 출하량, 수요 측 시그널(소비 데이터, 입찰 데이터 등)
  4. **대체재 위협**: 신기술, 신규 업태가 해당 업종에 주는 충격
  5. **업종 지수 흐름**: 동종 업종 종목의 최근 흐름, 해당 기업이 아웃퍼폼/언더퍼폼/동조 중인지
  6. 핵심 판단: **이것이 기업 개별 이벤트인가, 아니면 업종 전체의 베타 변동인가?**
  7. 각 이벤트에 출처와 날짜 표기

#### 작업 4: 시장 심리와 셀사이드/인플루언서 (sentiment-tracker)

- **subject**: `{회사명}의 최근 {N}일간 시장 심리와 기관 견해 변화 정찰`
- **description**:
  1. **셀사이드 등급 변동**: 골드만삭스, 모건, CICC 등의 최근 등급/목표가 조정
  2. **기관 보유 변화**: 13F 공시(미국주), 홍콩 후강퉁/선강퉁 보유, 북향자금(중국 본토→홍콩 자금) 흐름
  3. **공매도 데이터**: 공매도 비율, 신규 발표된 공매도 리포트(있을 경우)
  4. **인플루언서 견해**: `python3 ~/ai-berkshire/tools/xueqiu_scraper.py` 를 호출해 段永平(돤융핑) 등 인플루언서의 최근 관련 발언을 수집할 수 있음
     - 段永平(돤융핑) user_id: `1247347556`
     - 명령 예시: `python3 ~/ai-berkshire/tools/xueqiu_scraper.py --user-id 1247347556 --keywords {회사명},{종목코드} --output /tmp/dyp-{회사명}.md`
     - 해당 기업이 段永平(돤융핑)/李录(리루)의 관심 종목일 때만 호출하고, 아니면 건너뛰어 시간을 절약
  5. **소문과 찌라시**: 미디어가 확인하지 않은 소문, 소셜미디어 토론 핫이슈(쉐치우(雪球, 중국 투자 SNS)/X/Reddit)
  6. **기술적 시그널**: 핵심 지지선/저항선 도달 여부, 대량 거래 유무, 신용거래(융자융권) 이상 여부
  7. 핵심 판단: **펀더멘털 주도인가, 아니면 심리/수급 주도인가?**

### 5단계: 4개 Agent 병렬 실행

**반드시 동일한 메시지 안에서 Task 도구를 4번 병렬 호출한다.** 각 Agent 설정:
- `subagent_type`: `general-purpose`
- `run_in_background`: `true`
- `team_name`: `{회사명}-newspulse`
- `name`: 해당 역할명 (company-event-scout / regulatory-watcher / industry-peer-analyst / sentiment-tracker)

각 Agent의 prompt 템플릿:

```
당신은 {회사명} 뉴스 펄스 팀의 "{역할명}"으로, {정찰 차원} 차원의 최근 {N}일간 이벤트를 정찰합니다.

시간 범위: {시작 날짜} ~ {오늘 날짜}
주가 급변 배경: {사용자가 제공한 급변 정보, 없으면 "특정 급변 없음, 정기 점검"이라고 기재}
정보 가용성 등급: {A/B/C}

작업 #{작업 번호}을(를) 완수하세요: {작업 subject}

구체적 정찰 요구사항:
{작업 description 의 내용}

**정찰 방법**:
- 시의성 쿼리에는 WebSearch 를 우선 사용(키워드에 날짜나 "최근", "latest", "2026" 추가)
- 핵심 이벤트는 WebFetch 로 원본 출처(공시 원문, 실적 보고서, 규제 문서)를 정독
- 각 이벤트에 "독립 출처 검증"을 수행 — 소문은 최소 2개 독립 출처 필요
- **헤드라인 낚시에 속지 말 것**: 헤드라인과 본문이 일치하지 않는 이벤트는 "헤드라인 오해 소지"로 표기

**출력 형식(중요)**:
1. **핵심 발견**: 가장 중요한 이벤트 3~5건, 각 1~2문장
2. **완전한 이벤트 타임라인 표**(날짜 역순):
   | 날짜 | 이벤트 | 출처 | 주가 급변 상관성 | 지속성 |
3. **본 차원 원인규명 결론**: 정찰한 이벤트를 바탕으로 "이 차원이 주가 급변을 설명할 수 있는가? 확신도는?"에 답
4. **데이터 공백 선언**: 어떤 정보를 못 찾았는지, 어디에 의문점이 있는지, 무엇이 추가 정보를 기다려야 하는지
5. "사실"과 "추측"을 엄격히 구분하고, CLAUDE.md 의 객관성 원칙을 준수

**완료 후**:
1. TaskUpdate 로 작업을 completed 로 표시
2. SendMessage 로 완전한 정찰 보고서를 team-lead 에게 전송 (type: "message", recipient: "team-lead")
```

### 6단계: 실시간 진행 추적

- 정찰 보고서를 한 건 받을 때마다, 해당 차원의 핵심 발견 3건을 사용자에게 보여준다
- 4건 전부가 도착할 때까지 대기
- 전부 도착하면 SendMessage 로 4개 Agent에게 shutdown_request 를 전송

### 7단계: team-lead 종합 원인규명

4건의 정찰 보고서를 종합해 **급변 원인규명 보고서**를 산출한다(연구 보고서가 아니라 "판단"이 핵심).

---

#### 1. 한 줄 원인규명
> 한 문단(30~60자)으로 설명: 이번 주가 급변의 주원인 + 부차 원인 + 성격 판단(가치 이벤트/심리 변동/불명)

#### 2. 완전한 이벤트 타임라인 (4개 차원 통합)

날짜 역순으로 모든 차원의 이벤트를 통합한다.

| 날짜 | 차원 | 이벤트 | 출처 | 급변 원인 가중치 |
|------|------|------|------|-----------|
| 2026-04-30 | 기업 | XX | 링크 | 🔴 고 |
| 2026-04-29 | 업계 | XX | 링크 | 🟡 중 |
| 2026-04-28 | 심리 | XX | 링크 | ⚪ 저 |

가중치 범례: 🔴 고(단독으로 급변을 설명하기에 충분) / 🟡 중(일부 기여) / ⚪ 저(배경 노이즈)

#### 3. 급변 원인규명 표

| 후보 설명 | 증거 | 반증 | 확신도 | 지속성 |
|---------|------|------|------|--------|
| 예: 실적 미스 | 매출이 예상 대비 5% 미달, 매출총이익률 하락 | 일회성 요인에 대해 경영진의 설명 있음 | 고 | 단기 1~2주 |
| 예: 업종 베타 | 동종업계가 같은 기간 8% 하락 | 해당 종목 하락폭이 업종보다 뚜렷이 큼 | 중 | 업종과 동조 |

#### 4. 성격 판단 (핵심 결론)

하나에 체크한다.

- [ ] **가치 이벤트**: 펀더멘털에 실제 변화 발생(실적, 해자, 경영진, 종착점), 투자 논거 재검토 필요
- [ ] **심리/기술적 변동**: 펀더멘털은 변화 없음, 수급/심리/베타 주도, 기회 또는 노이즈로 볼 수 있음
- [ ] **진짜 원인 불명**: 주가 급변폭에 부합하는 이벤트를 찾지 못함 — **가장 위험한 결론**. 시장이 무언가를 미리 알고 있거나(내부정보/선행 매매), 우리가 정보원을 놓쳤거나 둘 중 하나
- [ ] **혼합**: 일부 가치 이벤트 + 일부 심리 증폭

#### 5. 각 차원 정찰 요약

차원마다 가장 중요한 발견 3~5건 + 해당 차원의 원인규명 기여도

#### 6. 행동 제안

| 행동 | 권장 여부 | 이유 |
|------|--------|------|

## 핵심 원칙

1. **속도가 완전함보다 우선** — 이 skill 의 핵심 가치는 10~15분 안에 원인규명 판단을 내리는 것이다. 심층 분석에 빠지지 말 것(그건 다른 skill 의 역할)
2. **나열보다 원인규명 우선** — 이벤트를 찾는 건 어렵지 않다. 어려운 건 "어떤 이벤트가 이번 주가 급변에 걸맞은지" 판단하는 것이다. 더하기가 아니라 빼기를 할 것
3. **"불명"을 솔직히 직면** — 주원인을 못 찾으면 "진짜 원인 불명"이라고 명확히 적는다. 인과 사슬을 억지로 끼워 맞추는 것보다 가치가 있다(시장이 악재를 선반영하고 있을 수 있다)
4. **입장을 예단하지 않음** — 보유 중이라는 이유로 "이건 심리 변동이니 괜찮다"는 쪽으로 기울지 말 것. 증거가 가리키는 방향대로 적는다
5. **"촉매"와 "우연"을 구분** — 같은 시점에 발생한 이벤트가 반드시 급변의 주원인은 아니다. 영향의 규모가 부합하는지 봐야 한다
6. **정보 가용성을 존중** — C급 기업은 정말로 아무 뉴스도 못 찾을 수 있다. 이 결론 자체를 반드시 적어야 한다
7. **`CLAUDE.md` 객관성 원칙 준수** — 모든 판단에 데이터 출처를 첨부하고, 사실과 견해를 구분한다
8. **사용자를 대신해 결정하지 않음** — 원인규명과 행동 제안 리스트는 제시하되, 매매 결정은 사용자가 한다
